## T1. Fix `axc` Log/Scale + Single YAML Interface

### Problem

`frame.axc.yscale: log` is not a reliable single source of truth. In practice, users often must also set layer-level `style.norm: LogNorm` to get expected colorbar behavior.

### Target

Make colorbar configuration explicit with one frame-level interface and minimal duplication.

### Proposed YAML Contract (v1.2.6)

```yaml
frame:
  axc:
    color:
      scale: log       # linear | log
      cmap: viridis
      vmin: 1
      vmax: 1e5
```

Compatibility rules:

- `frame.axc.color` is the preferred interface in v1.2.6.
- `frame.axc.color.scale` is the colorbar scale contract; legacy `axc.yscale/xscale` and layer `style.norm` were transitional inputs in this release window.
- If `color.scale=log`, runtime auto-builds `LogNorm` using `color.vmin/color.vmax` (or inferred limits).

### Acceptance Criteria

- `frame.axc.color.scale: log` alone can drive log colorbar (when data range is valid).
- `frame.axc.color.vmin/vmax` controls color-axis lower/upper bounds.
- Vertical/horizontal colorbars choose correct axis automatically.
- Missing `ticks_position` no longer raises errors; defaults are applied.
- Manual ticks, when specified, are actually applied.
- Existing YAML using `frame.axc.color.scale: log` remains functional.
